## Task 1.2: Добавить маршрут и обработчик страницы тега

### Связь с планом и ТЗ

- Фича: `blog_feature_tags`
- План: `agency/plans/blog_feature_tags_PLAN_example.md`, раздел «Задача 1.2».
- ТЗ: `agency/tz/blog_feature_tags_TZ_example.md`, UC-01.

### Цель задачи

Реализовать серверную часть страницы тега: маршрут и обработчик, который находит тег по `slug` и отдаёт связанные материалы в шаблон.

### Входные данные для разработчика

- Текущая архитектура и роутинг описаны в `agency/ARCHITECTURE.md`.
- Требования к структуре контента и SEO — `agency/CONTENT_RULES.md`.

### Требуемые изменения (пример формата)

- **Файлы для изменений** (уточнить реальные пути):
  - `app/routes.php` или эквивалент — добавить маршрут `/tag/{slug}/`.
  - `app/controllers/TagController.php` (если нет — создать) — метод `showTagPage(string $slug)`.

- **Логика обработчика**:
  1. Получить тег по `slug`.
  2. Если тег не найден:
     - вернуть 404 или другой сценарий, согласованный в ТЗ.
  3. Получить связанные материалы (проекты и посты блога).
  4. Отсортировать материалы по дате (новые сверху).
  5. Передать данные в шаблон страницы тега.

### Ожидаемый результат

- Маршрут `/tag/{slug}/` работает.
- При корректном `slug` отображается страница тега (шаблон будет сделан в отдельной задаче).
- При некорректном `slug` поведение соответствует ТЗ (404 или редирект).

### Тесты

- Юнит/интеграционные тесты на:
  - существующий тег с материалами;
  - тег без материалов;
  - несуществующий тег.
